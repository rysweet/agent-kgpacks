# Keep the Repository Clean of Orchestration Artifacts

This guide explains which files do **not** belong in the wikigr repository, why they are
excluded, and how `.gitignore` guards prevent them from being accidentally committed.

---

## Background

wikigr is often developed using AI-assisted orchestration tools (such as amplihack) that
generate temporary files in the working directory. These files are internal scaffolding for
the orchestration session and are not part of the wikigr project.

Two issues were filed when orchestration artifacts were inadvertently committed:

| Issue | File | Problem |
|-------|------|---------|
| [#247](https://github.com/rysweet/wikigr/issues/247) | `launcher.py` | Imported `amplihack.recipes`, a package absent from `pyproject.toml`, causing `ModuleNotFoundError` on clean installs |
| [#246](https://github.com/rysweet/wikigr/issues/246) | `run.sh` | Contained hardcoded `/tmp/amplihack-workstreams/ws-247` paths and `AMPLIHACK_TREE_ID` session tokens, leaking developer environment details to all repository readers |

A third file — `workstreams.json` — exposed internal workflow IDs and agent session state.

All three files have been removed from version control and the working tree. `.gitignore` now
prevents any of them from being staged in future sessions.

---

## Files That Must Never Be Committed

### `launcher.py`

Generated by amplihack as an entry point for the orchestration session. It imports from
`amplihack.recipes`, which is not a wikigr dependency and does not exist in a standard Python
environment. Committing this file breaks `pip install` and `uv sync` for any contributor who
does not have amplihack installed.

### `run.sh`

Generated by amplihack to start a specific orchestration session. It contains:

- Absolute paths such as `/tmp/amplihack-workstreams/ws-NNN` that are only valid on the
  developer's machine during that session.
- Session-specific environment variables (`AMPLIHACK_TREE_ID`, `AMPLIHACK_SESSION_ID`) that
  expose internal tooling state.

This file is not portable and constitutes information disclosure.

### `workstreams*.json`

The amplihack task list for a session. Contains internal workflow IDs, agent session metadata,
and the sequence of steps executed by the orchestrator. This is not a wikigr build artifact
and serves no purpose for wikigr contributors.

## How `.gitignore` Guards Work

Three glob patterns under the "Amplihack workstream orchestration files" section of `.gitignore`
prevent these files from ever being staged:

```gitignore
# Amplihack workstream orchestration files (not wikigr project artifacts)
launcher.py
run.sh
workstreams*.json
```

These patterns match at the repository root only (no leading `**/`), which is intentional:

- `launcher.py` and `run.sh` are generic names that could legitimately appear in
  subdirectories (e.g., a test fixture). The guard is scoped to the root to avoid false
  positives.
- `workstreams*.json` matches `workstreams.json`, `workstreams-2024-01-01.json`, and similar
  variants that amplihack may generate, without affecting nested JSON files in `data/packs/`.

## Checking for Stale Artifacts

After an orchestration session, verify no new artifacts were left behind:

```bash
# List any untracked files at the repo root
git status --short | grep '^\?\? '

# Confirm the three guarded patterns are ignored
git check-ignore -v launcher.py run.sh workstreams.json
```

Expected output for the second command (line numbers reflect the current `.gitignore`):

```
.gitignore:108:launcher.py	launcher.py
.gitignore:109:run.sh	run.sh
.gitignore:110:workstreams*.json	workstreams.json
```

If `git check-ignore` produces no output for a file, the pattern is missing from `.gitignore`
and the file could be accidentally staged.

## What `pyproject.toml` Contains (and Does Not Contain)

`pyproject.toml` declares only wikigr's own dependencies. Orchestration-layer packages such as
`amplihack` and `amplihack.recipes` are intentionally absent.

If you encounter a `ModuleNotFoundError` for a package not listed in `pyproject.toml`, the
import is from an orchestration artifact rather than wikigr source code. Remove the file that
contains the import rather than adding the package to `pyproject.toml`.

---

## Security Considerations

The `.gitignore` guards address three security concerns:

1. **Supply chain risk** — `launcher.py` imported a package (`amplihack.recipes`) not
   pinned in `pyproject.toml` or `uv.lock`. Any package not in the lock file bypasses
   integrity verification.

2. **Information disclosure** — `run.sh` exposed absolute local paths and session tokens.
   While these tokens have no value to an attacker after the session ends, absolute paths
   reveal developer machine layout and the specific orchestration tool and version in use.

3. **Metadata leakage** — `workstreams.json` revealed the number of work items, their
   descriptions, and the internal step sequence of the development session.

Removing these files and adding `.gitignore` guards eliminates all three risks for future
sessions.

**Future improvement:** A `check-banned-patterns.sh` pre-commit rule that rejects `/tmp/` and
`/home/` absolute path patterns in non-test Python files would catch issue #246-class commits
at the point of staging rather than relying on contributors to notice them during code review.
